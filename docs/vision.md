# Vision — Banking System

## Назва проєкту
**BankingSystem** — консольна банківська система на C#/.NET

## Опис проблеми
Невеликі кредитні спілки та навчальні фінансові установи потребують простого інструменту управління рахунками клієнтів, який дозволяє проводити основні банківські операції, відстежувати транзакції та нараховувати відсотки — без надмірної складності промислових систем.

## Цільові користувачі
- **Касир/оператор** — виконує повсякденні операції: відкриття рахунків, перекази, видача кредитів.
- **Менеджер** — переглядає виписки, стан кредитів, нараховує відсотки по всьому портфелю.
- **Адміністратор** — реєструє нових клієнтів, налаштовує параметри рахунків.

## Основні сценарії використання

### UC1 — Переказ між рахунками (головний сценарій, Lab 34)
1. Оператор обирає рахунок-відправник і вводить суму.
2. Система перевіряє достатність коштів (з урахуванням овердрафту для CheckingAccount).
3. Система дебетує відправника, кредитує отримувача.
4. Записуються дві транзакції (Withdrawal + Deposit), стан зберігається.
5. Оператор бачить підтвердження з новими балансами.

### UC2 — Нарахування відсотків по ощадних рахунках (Lab 35)
1. Менеджер запускає нарахування відсотків за поточний місяць.
2. Система добирає всі SavingsAccount з балансом > 1000 грн через LINQ.
3. InterestCalculator обраховує суму за обраною стратегією (проста/складна).
4. Транзакції типу Interest записуються на кожен рахунок.
5. Менеджер бачить зведений звіт.

### UC3 — Видача та виплата кредиту (Lab 35)
1. Оператор запитує кредит для клієнта: сума, термін, тип.
2. Система перевіряє скоринг: баланс клієнта > мінімального порогу, відсутність прострочених кредитів.
3. При схваленні — формується Loan з графіком LoanPayment, кошти зараховуються.
4. Щомісяця оператор виконує виплату: сума списується з рахунку, оновлює LoanPayment.
5. При повному погашенні кредит закривається.

### UC4 — Перегляд виписки по рахунку (Lab 35)
Фільтрація транзакцій за датою/типом через LINQ, формування підсумку.

## Нефункціональні вимоги

| # | Вимога | Обґрунтування |
|---|--------|---------------|
| NFR1 | **Тестованість** — жодна бізнес-логіка не залежить від конкретного репозиторію чи UI | Сервіси отримують `IRepository<T>` через конструктор → легко підміняти in-memory |
| NFR2 | **Стійкість до помилок** — некоректний стан об'єкта неможливий | Конструктори перевіряють інваріанти; операції кидають `DomainException` або повертають `Result<T>` |
| NFR3 | **Розширюваність** — новий тип рахунку або стратегія відсотків не ламає існуючий код | Відкрито для розширення через `abstract Account` і `IInterestCalculator`; закрито для модифікації (OCP) |
| NFR4 | **Читабельність** — код читається як бізнес-правила, не як технічний шум | Назви методів: `Deposit`, `Withdraw`, `Transfer`; не `UpdateBalance`, `SetValue` |
| NFR5 | **Персистентність** — стан зберігається між запусками | JSON-файли через `System.Text.Json`; Infrastructure відповідає за I/O |

## Обмеження першої ітерації (Lab 34)
Свідомо **не** реалізується:
- Реальна JSON-персистентність (лише in-memory репозиторій)
- UC2 (відсотки) та UC3 (кредити) — заплановано на Lab 35
- Автентифікація користувачів
- Асинхронний I/O
- Звіти та аналітика
